# Overview
I gathered insights on Cloud Spanner, so here are my notes. It's a rough draft, so it's not categorized.

# Notes
- No downtime for planned maintenance or schema changes, guaranteeing up to 99.999% availability.
  - Cloud SQL has downtime for planned maintenance.
- Architecture of Cloud Spanner
  - Client
    - Servers or containers where R/W code is executed.
    - Communication can be done via gRPC or REST API using client libraries.
  - Node
    - Compute resources that execute R/W operations.
    - Nodes do not have local storage (disks) attached.
    - Data is stored in a distributed file system (internal name Colossus).
      - Colossus is a cluster-level file system that is a successor to Google File System.
        - It seems to be used as a storage foundation for various services operated by Google.
  - Split
    - The unit of data in Cloud Spanner.
    - Managed by a specific node.
    - The capacity limit is 4GB; exceeding this generates a new Split, and the data is divided.
- Replication
  - Read/Write replicas
  - Read-only replicas
  - Witness replicas
    - Replicas that do not hold a complete copy of the database and only participate in voting.
- For areas that only perform READ operations, it is recommended to use Read-Only Transactions. They do not acquire locks, so they do not block other transactions. The contents of Read-Only Transactions should maintain idempotency to allow for retries.
- Even if hotspots occur, performance equivalent to one Node is achieved. Therefore, if running in a development or testing environment with one Node, it may be difficult to notice hotspots. Be cautious during load testing.
- Data is distributed not only by increasing or decreasing Nodes but also by generating Splits.
  - Splits are generated by load-based splits (due to increased write/read load) and size-based splits (due to increased data volume).
  - Keep this in mind when you want to warm up.
- Indexes are stored in tables just like regular tables.
  - Indexes can also be interleaved.
- Index generation takes time depending on the number of nodes and data volume (the most efficient time is during table creation).
- If specific key values are expected to concentrate data, it may be better not to create indexes from the start (as hotspots may occur).
- When adding indexes to existing tables with many records, be cautious about the number of indexes added per day (the documentation suggests around three is good).
  - Adding too many indexes can increase the system load on Spanner and degrade performance.
- Using the STORING clause allows additional columns to be stored in the index.
  - Particularly effective when there are many references. Be cautious as high write volumes can increase write costs.
- It seems better to refer to the query execution plan to decide whether to use FORCE_INDEX or a covering index.
- The order of index columns should ideally match the order in which you want to retrieve them to avoid unnecessary disk seeks.
- Interleaving is possible up to six levels.
- Interleaved child tables cannot have their interleaving removed.
- Counting mutations
  - The number of Column*row included in the transaction.
    - [Mutations per commit (including indexes): 20000](https://cloud.google.com/spanner/quotas?hl=en#limits_for_creating_reading_updating_and_deleting_data)
  - Be cautious when considering mutation limits.
  - Refer to [Google Cloud Spanner Deep Dive](https://docs.google.com/presentation/d/1XKaOrex3WS8xZ0TsjsTQKBgxjVSGxkyqdcIqG66zd64).

# Thoughts
If the design is wrong, you won't be able to leverage Spanner's scalability and achieve sufficient performance. The lack of downtime for planned maintenance and schema changes is a significant operational advantage. You need to be aware of data distribution from the perspective of Splits to maximize performance during operation. Since I've mostly been looking at Japanese articles, it might be good to read more articles from abroad.
