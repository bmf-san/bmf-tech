# 概要

アーキテクチャ設計における共通化には、気軽に進めてよいものと、慎重に避けるべきものがある。

ロギングや監視、認証基盤のような**技術的関心事**は、むしろ積極的に共通化したほうがよい。一方で、**特定のドメインに属するロジック**を「似ているから」という理由で共通化すると、後から引き剥がしにくい大きな罠になる。本稿はこの非対称性を、境界づけられたコンテキスト（Bounded Context）の観点から整理するものである。

なお、扱うのはマイクロサービスに特有の話ではない。物理的な分割（サービス分割）ではなく、**論理的な境界（境界づけられたコンテキスト）**の問題であり、モジュラモノリスやパッケージ分割でも同じ罠は起きる。

# 共通化しやすい技術的関心事

共通化が比較的安全に効く領域がある。ロギング、メトリクス収集、分散トレーシング、HTTP基盤、エラー通知、設定読み込みといった、いわゆる**技術的関心事(technical concerns)** である。

これらは次の性質を持つ。

- ビジネスの意味を直接には持たない
- 変更の駆動要因が基盤側にある（ライブラリのバージョンアップ、SREの運用要求など）
- 複数の業務文脈から使われても、文脈ごとに振る舞いを変える必要がほとんどない

横断的関心事(cross-cutting concern)として切り出し、共通ライブラリや基盤サービスとして独立進化させるのは合理的である。ここで重複を必要以上に許容すると、かえって全体の保守性を下げてしまうことが多い。

# 技術的関心事にドメインが染み出すグレーゾーン

ただし、「技術的関心事」と一括りにしてしまうと少し大雑把である。純粋に技術的な部分と、ドメイン知識が染み出している部分を区別しないと、別の形の罠を踏む。

代表例は**認証と認可**である。認証（誰であるかを検証する）は技術基盤として共通化しやすい。一方、認可（その人が何をしてよいか）は、「販売担当者は自部門の注文のみ参照できる」「与信担当者は承認フロー中の案件のみ閲覧できる」といった、明らかにドメインルールの塊である。これをまとめて「認証認可ライブラリ」に押し込むと、ドメイン変更のたびに基盤を触ることになり、基盤の変更速度と業務要件の変更速度の不整合が発生する。

ロギングも同様で、「どの項目を個人情報として伏せるか」「どのイベントを監査ログとして永続化すべきか」はドメイン知識である。共通ロガーはフォーマットと出力先だけを受け持ち、**何をどう扱うかの判断はドメイン側に残す**べきである。

技術的関心事は積極的に共通化してよいが、**その中に紛れ込んでいるドメイン判断を見逃さない**ことが重要になる。

# 共通化してはいけないドメインロジック

本稿で最も強調したいのはここである。**特定のドメインに属するロジックの共通化は、アーキテクチャ上の大きな罠になる**。

「顧客」「注文」「価格」のように、複数のサブシステムに同じ名前のモデルが登場することは珍しくない。構造も一見似ている。だからといって共通のモデルや共通のサービスにまとめると、時間が経つほど綻びを生みやすくなる。

DDDの言葉を借りれば、同じ言葉でも**境界づけられたコンテキストが異なれば別物**である。販売ドメインにおける「顧客」は、購買履歴と配送先を持ったエンティティかもしれない。与信ドメインにおける「顧客」は、信用スコアと審査状態を持ったエンティティである。マーケティングにおける「顧客」は、セグメントとキャンペーン応答履歴を持つ。名前と識別子が同じだけで、不変条件もライフサイクルも、変更の駆動要因も異なる。

# なぜドメイン共通化が罠になるのか

ドメインロジックを無理に共通化すると、いくつかの力学が働く。

- **文脈ごとの不変条件の衝突**：片方の文脈では必須な属性が、もう片方では不要。片方で許される状態遷移が、もう片方では禁止。共通モデルにまとめれば、すべての文脈で最も厳しい制約に従うか、フラグと分岐で塗り固めるかになる。
- **変更速度の律速**：共通化されたコンポーネントは、最も慎重に扱われる文脈の変更速度に引きずられる。本来なら独立に進化できたドメインが、互いに足を引っ張り合う。
- **所有者の消失**：Shared Kernel的（複数のコンテキストが同じドメインモデルを共有所有するDDDのパターン）に共有されたドメインコードは、複数チームのどこにも完全には属さなくなる。誰もが触れるが、責任の所在が曖昧になりやすい。Conway's Law的にいえば、組織境界を越えたコードは摩擦を生む。
- **剥がすコストの肥大化**：一度「共通ドメインモデル」として定着すると、それを前提に多くのコードが書かれる。後から文脈ごとに分離しようとすると、データ移行、API互換、テスト、チーム合意のすべてが絡む巨大な作業になる。

# 見分けるための問い

共通化の是非を判断する際、次の問いが有効である。

- **変更理由はどこにあるか**：変更が基盤都合（ライブラリ、運用、セキュリティ）で起きるか、ドメイン都合（業務ルール、商習慣）で起きるか
- **所有者は同じか**：コードを所有し、変更判断を下すチームと文脈は単一か
- **将来、文脈ごとに別の進化を遂げる可能性はあるか**：今は同じに見えても、業務要件の変化で分岐する可能性があるなら、共通化が将来の足枷になりやすい

「コードの形が似ている」ことは、共通化の十分条件ではない。似ているのが**偶然**なのか**本質**なのかを問う必要がある。

# 常に分離が正義とは限らない

ここまでドメイン共通化の危険を述べたが、どんな状況でも分離すべきだと言いたいわけではない。

プロダクトの初期フェーズや小さなモノリスにおいて、将来の境界づけられたコンテキストを予測して過剰に分離するのは、別種の罠である。ドメインの輪郭がまだ定まっていない段階では、むしろ一つの場所で進化させ、境界が見えてから切り出す方が安全なことが多い。

重要なのは、「分離する/しない」を一度決めたら終わりにせず、**組織規模・プロダクトフェーズ・ドメインの成熟度に応じて見直し続ける**ことである。

# まとめ

共通化は技術判断のように見えて、実際には組織とドメインに関する判断を含む。

- 技術的関心事は積極的に共通化してよい。ただしその中にドメインが染み出している部分を見極めること
- ドメインロジックは、名前と構造が似ているだけでは共通化の理由にならない。境界づけられたコンテキストが異なれば別物として扱う
- 判断軸は「コードの形が似ているか」ではなく、「**変更理由とドメイン文脈が同じか**」である

共通化は常に善ではない。安易な共通化が、アーキテクチャにとって静かに深い負債を生む。
